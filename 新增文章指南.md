# 新增文章完全指南

> 本文档的最终作用：让你或自动化助手在 `src/content/posts/` 中新增、更新文章时，能够稳定产出 **可构建、可收录、可阅读、可转化** 的 SEO 内容页。  
> 它不是单纯教你“怎么写一篇文章”，而是把每篇文章都做成一个小型引流入口：承接搜索需求 → 帮用户做判断 → 通过合规 CTA 引导用户主动进入小店。

---

## 0. 一句话流程

**选题 → 判断搜索意图 → 新建 Markdown → 填写 Frontmatter → 按模板写正文 → 检查内链和 CTA → `git push` → 自动构建、SEO 检查、部署、搜索引擎通知。**

推送后自动流程会尝试完成：

```text
文章入库 → Astro 静态构建 → URL 导出 → SEO 检查 → 链接检查 → 搜索引擎通知 → Cloudflare Pages 部署
```

你真正要保证的是：**文章内容和格式符合本文档规则**。

---

## 1. 项目目标与文章定位

这是一个 AI 工具选型 SEO 内容站，目标是通过搜索流量引导用户进入小店查看当前可选方案。

文章不要写成“百科介绍”，而要写成“购买前决策辅助页”。

每篇文章都应该完成 4 件事：

1. **接住搜索需求**：标题、描述、正文前 300 字要明确回应用户搜索的问题。
2. **降低选择成本**：用场景、预算、使用频率、适合/不适合人群帮助用户快速判断。
3. **建立信任感**：不夸大、不承诺效果、不写虚假价格/库存/评分。
4. **自然引导转化**：在用户完成判断的节点，给出小店入口，而不是硬插广告。

## 2. 新文章完整流程

### 2.1 确定文件名

在 `src/content/posts/` 下新建 Markdown 文件。

文件名就是 URL slug，规则如下：

| 规则 | 要求 |
|---|---|
| 语言 | 使用英文 |
| 格式 | 使用连字符 `-`，即 kebab-case |
| 长度 | 简短清晰，建议 3-7 个单词 |
| 关键词 | 尽量包含核心搜索词 |
| 后缀 | 必须是 `.md` |

示例：

```text
src/content/posts/claude-pro-vs-chatgpt-plus.md
src/content/posts/ai-tools-for-teachers.md
src/content/posts/chatgpt-plus-buying-guide.md
```

不要使用：

```text
新文章.md
ChatGPT对比Claude.md
chatgpt_vs_claude.md
chatgpt vs claude.md
```

---

### 2.2 slug 必须和文件名一致

如果文件名是：

```text
claude-pro-vs-chatgpt-plus.md
```

那么 Frontmatter 里的 `slug` 必须是：

```yaml
slug: "claude-pro-vs-chatgpt-plus"
```

最终页面地址会是：

```text
/claude-pro-vs-chatgpt-plus/
```

## 3. Frontmatter 标准模板

每篇文章开头必须包含 Frontmatter。所有字段都要填写。

### 3.1 可直接复制的模板

```yaml
---
title: "文章标题：必须包含核心搜索关键词"
description: "文章描述建议 120-160 字，说明本文会解决什么问题，适合哪些人阅读，并自然包含核心关键词和相关变体。"
slug: "your-article-slug"
category: "工具选择"
primaryKeyword: "用户最可能搜索的核心词"
secondaryKeywords:
  - "长尾搜索词1"
  - "长尾搜索词2"
  - "长尾搜索词3"
intent: "tool"
updatedAt: "2026-06-08"
ctaText: "查看当前可选 AI 工具方案"
faq:
  - question: "FAQ问题1：最好包含主关键词？"
    answer: "直接回答，不要绕圈，控制在 40-120 字。"
  - question: "FAQ问题2？"
    answer: "回答用户购买前、选择前、使用前最关心的问题。"
  - question: "FAQ问题3？"
    answer: "避免承诺效果，避免虚假价格、库存、评分。"
  - question: "FAQ问题4？"
    answer: "可以引导用户按用途、预算和频率判断。"
  - question: "FAQ问题5？"
    answer: "最后一个问题可以承接到小店入口，但不要写成硬广告。"
---
```

---

### 3.2 字段约束速查表

| 字段 | 是否必填 | 要求 | 作用 |
|---|---:|---|---|
| `title` | 是 | 包含核心搜索关键词，不能和其他文章 H1 重复 | 页面 `<title>` 和 H1 |
| `description` | 是 | 建议 120-160 字，包含关键词和用户收益 | 搜索摘要和页面描述 |
| `slug` | 是 | 必须和文件名一致，不含 `/` | 页面 URL |
| `category` | 是 | `工具选择` / `工具对比` / `场景需求` | 文章分类和面包屑 |
| `primaryKeyword` | 是 | 一个核心搜索词 | 内容定位 |
| `secondaryKeywords` | 是 | 至少 1 个，建议 3-6 个 | 长尾词覆盖 |
| `intent` | 是 | `tool` / `comparison` / `scenario` / `guide` | 搜索意图分类 |
| `updatedAt` | 是 | `YYYY-MM-DD` 格式 | 更新时间和 sitemap lastmod |
| `ctaText` | 是 | 根据文章场景写，不要所有文章一样 | 顶部 CTA 按钮文案 |
| `faq` | 是 | 恰好 5 个问答 | FAQ 结构化内容 |

## 4. 搜索意图与文章类型

写文章前先判断用户为什么搜索这个词。搜索意图决定标题、正文结构和 CTA 文案。

| intent | 适用文章 | 用户心态 | 文章重点 | CTA 方向 |
|---|---|---|---|---|
| `tool` | 单个工具值不值得买 | 已经知道工具名，想判断是否适合自己 | 适合谁、不适合谁、使用场景、替代方案 | 查看该工具相关方案 |
| `comparison` | A 和 B 怎么选 | 在两个或多个工具之间犹豫 | 对比表、选择建议、场景分流 | 按用途进入小店选择 |
| `scenario` | 某场景用什么 AI 工具 | 有明确需求，但不知道工具 | 场景拆解、推荐组合、预算建议 | 查看该场景工具方案 |
| `guide` | 购买指南、避坑指南、低成本方案 | 准备购买或想降低风险 | 决策流程、注意事项、常见误区 | 查看当前可选分类 |

## 5. 标题、描述与首段写法

### 5.1 标题公式

标题要让用户一眼知道“这篇能帮我做决定”。

常用公式：

```text
工具名 + 值不值得买 / 怎么选 / 适合谁
A 和 B + 怎么选 / 区别 / 对比
场景 + 用什么 AI 工具 / 推荐方案
人群 + AI 工具推荐 / 低成本方案
问题 + 购买指南 / 避坑指南
```

示例：

```text
ChatGPT Plus 现在还值得买吗？适合人群、使用场景和购买前提醒
Claude Pro 和 ChatGPT Plus 怎么选？写作、办公和编程场景对比
学生党用什么 AI 工具？低成本组合和购买前注意事项
```

---

### 5.2 Description 写法

`description` 不是随便写一句简介，它会影响搜索结果点击率。

建议结构：

```text
本文围绕【核心关键词】，从【用户关心的维度1】、【维度2】、【维度3】进行分析，帮助【目标人群】判断是否适合自己，并整理购买前需要注意的问题。
```

示例：

```text
本文围绕 ChatGPT Plus 是否值得买，从适合人群、办公学习场景、与 Claude/Gemini 的区别、购买前注意事项等角度分析，帮助第一次选择 AI 工具的用户做出更稳妥的判断。
```

---

### 5.3 首段必须快速回答问题

正文开头不要铺垫太久。前 300 字应直接回答：

- 这篇文章解决什么问题；
- 适合谁看；
- 简短结论是什么；
- 后文会从哪些角度判断。

推荐开头：

```markdown
很多人在选择 AI 工具时，会先问“ChatGPT Plus 现在还值得买吗”。如果你主要用于办公写作、学习提问、代码理解和日常效率提升，它仍然是比较通用的选择；但如果你更重视长文写作、资料整理或搜索核验，也需要同时对比 Claude、Perplexity 等工具。

本文不会只列功能，而是按使用场景、预算、使用频率和购买前注意事项，帮你判断是否适合自己。
```

## 6. 正文硬性指标

构建时会自动检查部分指标，不达标可能导致构建失败。

| 指标 | 最低要求 | 目的 |
|---|---:|---|
| 中文字数 | ≥ 1200 字 | 避免低质量薄内容 |
| CTA 购买链接 | ≥ 3 个 | 保证转化入口充足 |
| 站内链接 | ≥ 5 个 | 提高站内流转和主题相关性 |
| H1 标题 | 恰好 1 个 | 页面结构清晰 |
| H1 唯一性 | 不能和其他文章重复 | 避免页面主题冲突 |
| 图片 alt | 所有图片必须有 alt | 基础可访问性和 SEO |

重要提醒：文章 Markdown 正文中不要再写 `# 一级标题`。页面 H1 会由 `title` 自动生成。正文从 `## 二级标题` 开始写。

## 7. 推荐正文结构

每篇文章不一定完全一样，但建议使用下面结构。这样最适合自动化批量生产，也更容易提高阅读完成率和点击转化。

```markdown
## 先说结论：这类用户适合，另一类用户不适合

用 2-4 段快速给出结论。

## 适合人群与不适合人群

| 人群 | 是否适合 | 原因 | 购买前提醒 |
|---|---|---|---|
| ... | ... | ... | ... |

[根据你的使用场景进入小店查看方案](小店链接)

## 核心场景对比

| 使用场景 | 推荐方向 | 判断标准 | 注意事项 |
|---|---|---|---|
| ... | ... | ... | ... |

## 和相关工具怎么选

插入 2-4 个站内链接，引导用户继续阅读相关页面。

## 购买前需要注意什么

列出风险提醒、适合/不适合情况，不夸大。

[已经明确需求？进入小店按分类查看](小店链接)

## 常见问题

正文中可简要承接 FAQ，不需要重复 frontmatter 的全部内容。

## 总结：按用途、预算和频率做选择

最后总结，并给出最终 CTA。

[确认需求，查看当前可选 AI 工具方案](小店链接)
```

## 9. CTA 购买链接规范

CTA 的作用不是“强行卖”，而是在用户完成判断后，给出下一步入口。

### 9.1 推荐使用 HTML CTA

SEO 检查脚本会统计 `data-shop-cta`，所以正文内 CTA 推荐使用 HTML 写法：

```html
<a href="https://pay.ldxp.cn/shop/ai007?utm_source=seo&utm_medium=organic&utm_campaign=ai_tools&utm_content=your-article-slug"
   target="_blank"
   rel="nofollow sponsored noopener"
   data-shop-cta
   data-page-slug="your-article-slug"
   data-cta-position="middle">
   根据你的使用场景进入小店查看方案
</a>
```

必须修改：

- `utm_content=your-article-slug`
- `data-page-slug="your-article-slug"`
- 按位置修改 `data-cta-position`，如 `top`、`middle`、`bottom`
- 按文章场景修改按钮文字

---

### 9.2 CTA 数量与位置

每篇文章正文中至少放 3 个 CTA。推荐位置：

| 位置 | 放置节点 | 用户心理 |
|---|---|---|
| 第 1 个 | 首个关键结论之后 | 用户已经知道大方向 |
| 第 2 个 | 对比表或场景表之后 | 用户开始形成选择偏好 |
| 第 3 个 | 总结段之后 | 用户准备采取下一步行动 |

不要只在文章开头和结尾放 CTA。最佳做法是：**每当你帮用户完成一次判断，就给一个下一步入口。**

---

### 9.3 CTA 文案要场景化

不要所有文章都用“进入AI工具小店选购”。

| 文章类型 | 推荐文案 |
|---|---|
| ChatGPT 文章 | 查看 ChatGPT Plus 当前可选方案 |
| Claude 文章 | 查看 Claude Pro 相关方案 |
| Cursor 文章 | 查看适合编程的 AI 工具方案 |
| 对比文章 | 按你的用途进入小店选择 |
| 论文写作 | 查看写作类 AI 工具方案 |
| 学生党 | 查看学生党可用方案 |
| 办公场景 | 查看办公提效 AI 工具方案 |
| 通用导购 | 前往小店按用途查看分类 |

---

### 9.4 CTA 前的引导句

CTA 前最好有一句自然承接，减少突兀感。

推荐写法：

```markdown
如果你已经明确自己主要用于写作、办公还是编程，可以先按用途查看当前可选方案，再决定是否需要单独购买某一个工具。

<a href="..." data-shop-cta ...>按用途进入小店查看方案</a>
```

不要这样写：

```markdown
赶紧购买，马上下单。
```

## 10. 站内链接规范

每篇文章正文中至少插入 5 个站内链接，所有站内链接必须以 `/` 开头。

站内链接的目的：

1. 帮用户继续比较；
2. 增加站内停留；
3. 强化主题相关性；
4. 让新文章接入已有内容矩阵。

### 10.1 内链放置原则

| 位置 | 链接类型 | 示例 |
|---|---|---|
| 讲单个工具时 | 链接到对比页 | `/chatgpt-vs-claude/` |
| 讲场景时 | 链接到工具页 | `/chatgpt-plus-worth-it/` |
| 讲预算时 | 链接到低成本方案 | `/ai-tools-low-cost-plan/` |
| 讲第一次购买时 | 链接到购买指南 | `/first-ai-membership-guide/` |
| 讲搭配使用时 | 链接到组合指南 | `/ai-tools-combination-guide/` |

### 10.2 内链写法

```markdown
如果你还在比较不同工具，可以继续看 [ChatGPT 和 Claude 怎么选](/chatgpt-vs-claude/)。
```

不要写成裸链接：

```markdown
/chatgpt-vs-claude/
```

## 12. 表格、列表和阅读体验规范

用户通常不是来“欣赏文章”的，而是来快速做选择的。正文要降低阅读成本。

### 12.1 每篇文章至少放 1-2 个表格

推荐表格类型：

```markdown
| 使用场景 | 推荐工具方向 | 适合人群 | 购买前提醒 |
|---|---|---|---|
| 办公写作 | ChatGPT / Claude | 经常写文档的人 | 先确认是否高频使用 |
| 资料搜索 | Perplexity | 需要查资料的人 | 注意核验来源 |
```

### 12.2 段落不要太长

建议：

- 每段 2-4 行；
- 一个小节只解决一个问题；
- 多用“适合 / 不适合 / 注意事项 / 结论”；
- 不要连续堆砌工具功能。

### 12.3 多写判断句，少写空泛介绍

推荐：

```markdown
如果你每天都会用 AI 写邮件、改文案、整理资料，ChatGPT Plus 的价值会更明显。反过来，如果你只是偶尔提问，免费版本或低成本组合可能更适合先试用。
```

不推荐：

```markdown
ChatGPT 是一款非常强大的人工智能工具，拥有很多功能，可以帮助用户提升效率。
```

## 13. FAQ 写法

每篇文章 Frontmatter 中必须有 5 个 FAQ。

### 13.1 FAQ 问题来源

优先写用户购买前会问的问题：

- 值不值得买？
- 适合新手吗？
- 和另一个工具怎么选？
- 学生党/程序员/办公用户适合吗？
- 是否需要多个工具一起用？
- 购买前要注意什么？

### 13.2 FAQ 答案要求

| 要求 | 说明 |
|---|---|
| 直接 | 第一两句先回答问题 |
| 具体 | 结合场景、预算、频率 |
| 克制 | 不承诺效果，不写虚假信息 |
| 可转化 | 可以自然提醒用户按用途查看方案 |

示例：

```yaml
- question: "ChatGPT Plus 适合第一次购买 AI 工具的人吗？"
  answer: "如果你主要用于办公、学习、写作和日常问答，ChatGPT Plus 通常是比较通用的选择。但如果你更重视长文写作或资料搜索，也建议同时对比 Claude 和 Perplexity。"
```

## 14. 禁止项与合规要求

### 14.1 禁止出现的高风险词

文章正文、标题、描述和 FAQ 中不要出现以下表达：

```text
破解、绕过、共享账号、低价卡密、卡密秒发、不封号、保号、灰产、平台不让发、薅羊毛
```

这些词可能导致 SEO 检查失败，也会降低站点合规性和长期稳定性。

### 14.2 禁止写法

| 禁止写法 | 原因 | 替代写法 |
|---|---|---|
| 保证稳定使用 | 过度承诺 | 建议以页面展示和实际可用情况为准 |
| 全网最低价 | 虚假价格风险 | 当前可选方案以小店页面为准 |
| 永久可用 | 过度承诺 | 购买前确认使用周期和说明 |
| 官方授权 | 未核实时不要写 | 按小店页面说明判断 |
| 自动跳转购买 | 用户体验差且不合规 | 由用户主动点击进入小店 |

### 14.3 不要复制小店商品全集

本站只做选型、教程、对比和购买前提醒，不复制小店完整商品信息，不展示虚假价格、库存、评分或评论。

商品相关信息以小店页面为准。

## 18. 常见构建失败原因

| 报错方向 | 常见原因 | 解决方法 |
|---|---|---|
| missing title | Frontmatter 缺少 `title` | 补全标题 |
| missing description | 缺少 `description` | 补全描述 |
| missing canonical | 页面路径或 SEO 配置异常 | 检查 slug 和路由 |
| expected one H1 | 正文写了 `#` 或没有标题 | 正文不要写 H1 |
| duplicate H1 | 标题和已有页面重复 | 修改 `title` |
| fewer than 1200 Chinese chars | 内容太短 | 增加场景分析、表格说明、购买前提醒 |
| fewer than 3 CTA links | CTA 不足或缺少 `data-shop-cta` | 补足 HTML CTA |
| fewer than 5 internal links | 内链不足 | 从速查表补 5 个相关链接 |
| forbidden term | 出现禁止词 | 删除或改写 |
| sitemap missing | 构建产物异常 | 重新构建并检查配置 |

## 20. 最终判断标准

一篇合格的新文章，不是“字数够了就行”，而是要满足以下标准：

1. 用户从搜索结果点进来，能在前 30 秒知道答案；
2. 用户读到中间，能通过表格完成初步判断；
3. 用户有继续比较需求时，能通过内链进入相关文章；
4. 用户准备采取行动时，能看到自然、明确、合规的小店入口；
5. 自动化构建、SEO 检查、链接检查都能通过；
6. 后续通过 UTM 和点击事件能分析哪篇文章、哪个 CTA 位置更有效。

按这份指南写文章，目标不是堆内容，而是让每篇文章都成为稳定的搜索流量入口和小店转化入口。
